Case study

Northstar Analytics: a calmer dashboard for faster operational decisions.

Northstar needed a dashboard that helped managers understand performance quickly without forcing them through dense tables and scattered status panels.

Role

UX structure, visual design direction, front-end implementation, responsive behavior, and interaction polish.

Timeline

Four-week sprint covering discovery, wireframes, interface design, coded prototype, and refinement.

Stack

Semantic HTML, CSS Grid, custom properties, accessible form controls, and lightweight JavaScript.

Challenge

The old dashboard showed data, but it did not create understanding.

Managers had to scan multiple disconnected modules to answer simple questions: what changed today, what needs attention, and which team should act first? The interface had too many competing weights, inconsistent spacing, and no clear priority system.

The goal was not to add more features. It was to make existing information easier to read and act on during busy workdays.

Goals

What the redesign needed to accomplish.

Clarify priority

Move important daily signals to the top and reduce visual competition between metrics.

Improve scanning

Use consistent cards, labels, spacing, and chart groupings so users can compare information quickly.

Support action

Connect alerts and summaries to next steps instead of leaving users with raw numbers.

Stay flexible

Create a front-end structure that could support additional modules without redesigning the whole page.

Wireframe and concept

The first wireframes focused on order, not decoration. I tested a top-level health summary, a focused alerts column, and a grid of supporting metrics. This made the page feel less like a report archive and more like a daily command center.

Design decisions

I kept the background white, used neutral cards, gave important numbers a consistent accent treatment, and separated modules with subtle borders instead of heavy color blocks. The interface felt lighter while still carrying dense information.

Development decisions

The build used semantic regions, CSS custom properties, grid-based layouts, accessible focus states, and small JavaScript enhancements for panel behavior. This kept the prototype easy to review and extend.

Features

Key metrics, trend cards, alert summaries, team performance modules, responsive table behavior, filter controls, and a compact mobile layout for quick checks.

Results

A clearer experience with less cognitive friction.

  • Managers could identify priority issues faster from the opening viewport.
  • The redesigned card system made future metrics easier to add consistently.
  • Mobile views became useful for quick status checks instead of feeling like a squeezed desktop page.
  • The team reported fewer clarification questions during daily review meetings.

Lessons learned

Dashboard design improves when it starts with decisions, not data. The most useful question was: what should this information help someone do next?

Next steps

The next iteration would add saved views, improved chart comparisons, deeper accessibility testing, and role-specific dashboard presets.
